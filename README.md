# Тестовое задание: Тестирование API заявок

## 1. Краткий тест-план

### Что проверяем

| Область | Содержание |
|---------|------------|
| **Создание заявки** | Валидация полей, обязательность, граничные значения, статус `new` по умолчанию |
| **Редактирование заявки** | Разрешённые поля, запрет на изменение `status`, поведение в разных статусах |
| **Статусная модель** | Разрешённые и запрещённые переходы для администратора |
| **Авторизация и доступ** | Изоляция заявок пользователей, права администратора |
| **Обработка ошибок** | Осмысленные сообщения, корректные HTTP-коды |
| **Граничные значения** | Предельные длины строк, границы `age` |

### Виды тестирования

- **Функциональное** — проверка всех правил создания и редактирования
- **Позитивное/негативное** — корректные и некорректные входные данные
- **Граничных значений** — `name` (1, 2, 50, 51), `age` (17, 18, 65, 66), `comment` (500, 501)
- **Тестирование статусной модели** — все разрешённые и запрещённые переходы
- **Тестирование безопасности** — изоляция данных, подмена ID, эскалация привилегий
- **API-тестирование** — через RestAssured
- **Автоматизированное регрессионное** — smoke-сьют на критический путь

### Риски

| Риск | Вероятность | Влияние | Митигация |
|------|-------------|---------|-----------|
| Race condition при смене статуса | Средняя | Высокое | Тесты на параллельные запросы |
| Утечка чужих заявок через перебор ID | Средняя | Критичное | Тесты безопасности |
| Неинформативные сообщения об ошибках | Высокая | Низкое | Проверка тела ошибки |
| Инъекции в текстовые поля | Низкая | Критичное | Негативные тесты с XSS/SQL |
| Отсутствие rate limiting | Средняя | Среднее | Нагрузочное тестирование |

## 2. Чек-лист

### Создание заявки (`POST /api/tickets`)

- [ ] Создание с полями `name` + `phone` (минимальный набор) → `201`
- [ ] Создание со всеми полями → `201`
- [ ] Автоматическое присвоение `status = "new"`
- [ ] Попытка передать `status` при создании → игнорируется/ошибка
- [ ] `name = "Ab"` (2 символа) → `201`
- [ ] `name = "A"` (1 символ) → `422`
- [ ] `name = 50 символов` → `201`
- [ ] `name = 51 символ` → `422`
- [ ] `name` пустое / `null` / отсутствует → `422`
- [ ] `phone` пустое / `null` / отсутствует → `422`
- [ ] `phone` невалидный формат → `422`
- [ ] `email` пустой → `201`
- [ ] `email` валидный → `201`
- [ ] `email` невалидный → `422`
- [ ] `comment = 500 символов` → `201`
- [ ] `comment = 501 символ` → `422`
- [ ] `age = 18` → `201`
- [ ] `age = 65` → `201`
- [ ] `age = 17` → `422`
- [ ] `age = 66` → `422`
- [ ] `age = "строка"` → `422`
- [ ] Создание без авторизации → `401`

### Редактирование заявки (`PATCH /api/tickets/{id}`)

- [ ] Редактирование `name` в статусе `new` → `200`
- [ ] Редактирование `phone` в статусе `new` → `200`
- [ ] Редактирование `email` в статусе `new` → `200`
- [ ] Редактирование `comment` в статусе `new` → `200`
- [ ] Редактирование `age` в статусе `new` → `200`
- [ ] Попытка изменить `status` пользователем → `422`/игнорируется
- [ ] Редактирование в статусе `in_progress` → `200`
- [ ] Редактирование в статусе `done` → `200`
- [ ] Редактирование в статусе `cancelled` → `422`
- [ ] Редактирование чужой заявки → `403`/`404`
- [ ] Частичное обновление (только одно поле) → `200`, остальные не изменились
- [ ] Валидация полей при редактировании работает как при создании

### Статусная модель (`PATCH /api/tickets/{id}` админом)

- [ ] `new → in_progress` → `200`
- [ ] `new → cancelled` → `200`
- [ ] `new → done` → `200` (если разрешён прямой переход)
- [ ] `in_progress → done` → `200`
- [ ] `in_progress → cancelled` → `200`
- [ ] `done → cancelled` → `200`
- [ ] `done → new` → `422`
- [ ] `done → in_progress` → `422`
- [ ] `cancelled → любой статус` → `422`
- [ ] Попытка смены статуса пользователем → `403`/`422`

### Безопасность и доступ

- [ ] Пользователь не видит чужие заявки (`GET`) → `403`/`404`
- [ ] Пользователь не редактирует чужие заявки (`PATCH`) → `403`/`404`
- [ ] Админ видит все заявки → `200`
- [ ] Неаутентифицированные запросы → `401`

## 4. Баг-репорты

### Баг-репорт #1

| Поле | Описание |
|------|----------|
| **ID** | BUG-001 |
| **Заголовок** | Администратор может перевести заявку из `done` обратно в `new` |
| **Приоритет** | Critical |
| **Серьёзность** | Blocker |
| **Окружение** | Staging, build 2.4.1, 19.06.2026 |
| **Шаги воспроизведения** | 1. Авторизоваться как `admin`<br>2. Создать заявку через `POST /api/tickets`<br>3. Перевести заявку в `done`: `PATCH /api/tickets/42 {"status": "done"}`<br>4. Попытаться вернуть в `new`: `PATCH /api/tickets/42 {"status": "new"}` |
| **Фактический результат** | Статус меняется на `new`, код `200` |
| **Ожидаемый результат** | `422 Unprocessable Entity`, тело: `{"error": "Нельзя перевести заявку из done в new"}` |
| **Вложения** | `curl -X PATCH https://api.staging.example.com/api/tickets/42 -H "Authorization: Bearer admin_xxx" -H "Content-Type: application/json" -d '{"status": "new"}'` → `200 OK` |
